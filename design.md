# 基本設計・詳細設計 — cis-pixel-optics-sim

## 1. 全体アーキテクチャ

```text
[ブラウザ]
   │ http://localhost:8000
   ▼
[アプリ]  Python 3.12（単一のconda/miniforge環境、pymeep同居）
   ├─ Web UI（FastAPI + HTML/CSS/JavaScript）
   ├─ ジョブ管理（実行キュー・進行状況・履歴）
   ├─ 結果解析・グラフ生成（matplotlib）
   │
   │ subprocess（JSONファイル受け渡し）
   ▼
   └─ FDTDワーカー（構造生成 → FDTD実行 → 結果JSON/NPZ出力）
```

- **Pythonバージョン**: 本プロジェクトは Python 3.12 で統一する
  （Pythonバージョンルール（3.14.5以上）の本プロジェクト限定例外。ユーザー承認 2026-07-12）。
  Meep（pymeep）が Python 3.12 に対応しているため、アプリ本体と計算エンジンを
  同一環境に同居させられる。
- FDTDワーカーは同一環境内でもサブプロセスとして起動する。長時間計算中も
  Web UIの応答を保ち、計算の中断（キャンセル）を可能にするため。
  受け渡しは JSON（入力パラメータ）と NPZ/JSON（結果データ）のファイル経由とする。

## 3. FDTD計算設計（engine）

### 3.1 計算領域と境界条件

- 単位系: Meepの慣例に従い µm 基準（a = 1 µm）
- 横方向（X, Y）: 周期境界。斜入射時はBloch周期境界で入射角を表現する
  - 1画素レンズ: 1画素分を計算セルとする
  - 2画素共有: 2×1画素、4画素共有: 2×2画素を計算セルとする
  - クロストーク評価時は3×3画素（中央照射）に拡張する
- 縦方向（Z）: 上下にPML（吸収境界）
- 光源: 平面波（ガウシアン時間波形）。CRAは波数ベクトルの傾きで与える

### 3.2 DTI（画素間分離）のモデル化

- Si内の画素境界に沿って、格子状の溝（トレンチ）を配置する
- 溝はSi上面（受光面側）から指定深さまで掘り、指定材料（初期値: SiO2、n≈1.46）で充填する
- パラメータ: あり／なし、幅（0.05〜0.3 µm）、深さ（0〜Si厚。Si厚と同値でフルDTI）
- 共有OCL（2画素/4画素）の場合、配置を「全画素境界」／「共有単位境界のみ」から選択できる
  （共有画素間のDTI有無は集光スポット直下の構造となるため、結果への影響が大きい）
- 2Dモードでは断面に現れる縦溝として、3Dモードでは格子として生成する
- 幅がメッシュ解像度より細い場合は実行前に警告する（初期解像度10 nm刻みでは幅0.05 µmは5セル）

### 3.2b 位置ずれ（オフセット）の扱い

- OCL偏心（`ocl.offset_um`）: レンズのフットプリント（底面の範囲）は画素に
  固定したまま、頂点位置だけを+X方向へずらす（形状が非対称になる）。
  頂点の左右をそれぞれ別の半幅を持つプロファイルでつなぐ。範囲はレンズ半幅未満
  （2026-07-13にユーザー指摘で「レンズ全体の平行移動」から再定義）
- DTIオフセット（`dti.offset_um`）: DTI格子のみを+X方向へずらす。
  範囲は±画素サイズ/2。OCLとDTIの相対的な位置ずれはこちらで評価する
- PD（フラックスモニター）と画素境界はどちらのオフセットでも動かない
- 周期境界セルでは、はみ出した形状の周期像が自動では作られないため、
  DTIオフセット指定時は±セル幅にずらした複製ジオメトリを置いて周期像を再現する
  （OCL偏心はフットプリント固定でセルからはみ出さないため複製不要）
- 両オフセットはスイープ対象に指定できる
- 3Dモードにも対応（2026-07-19 追加）: 偏心はX方向（2D断面と同じ向き）に
  頂点をずらす。DTIオフセットはX方向の壁だけを+Xへずらし、Y方向の壁は
  全幅にわたるため位置は変わらない。周期像の複製は2Dと同じ扱い

### 3.2c OCLギャップ高さ（傾きレンズ）

- レンズ端（隣のレンズとのギャップ部分）の材料高さを左右個別に指定できる
  （`ocl.gap_height_left_um` / `gap_height_right_um`、0〜レンズ高さ未満）
- レンズ面は「左端の高さ→頂点→右端の高さ」を偏心と同じ片側別プロファイルで
  つなぐ。左右で違う値にするとレンズが傾いた状態の集光を評価できる
- 端の高さが0でない側は、ポリゴンに底面までの垂直な側面を追加する
- 両方ともスイープ対象に指定できる
- 3Dモードにも対応（2026-07-19 追加）: 各スライスのX方向範囲は2Dと同じ
  非対称プロファイルの逆算（`lens_slice_bounds_x_um`）で求める。
  Y方向の端の高さには左右ギャップ高さの平均を使う近似
  （左右差による傾きはX方向にのみ現れる）。偏心0・ギャップ0のときは
  従来のスライス生成と完全一致する（回帰確認済み）

### 3.2d OCL混在パターン（F-13）

- `ocl.pattern`（例: `["single", "shared2", "single"]`）でレンズを左から並べる。
  指定時は共有方式（`ocl.sharing`）より優先。未指定なら従来どおり1種類
- レンズ1枚の幅は single=1画素、shared2/shared4=2画素（2D断面のためshared4は近似）
- セル幅は合計画素数×ピッチ。Bloch周期境界により、このパターンが無限に
  繰り返される配列として計算される
- DTI「共有単位境界のみ」はレンズ単位の境目にだけ溝を置く
  （`pixel_boundary_positions_mixed_um`）
- 偏心・ギャップ高さは全レンズに同じ値が適用される。偏心の上限は
  パターン中で一番小さいレンズの半幅を基準に検証する
- 最大レンズ数は `MAX_PATTERN_UNITS`（8枚）。2Dのみ対応
- 受光内訳の評価（中央照射）と併用できる（2026-07-19 追加）:
  パターンのレンズ列をそのまま置き（横方向はPMLで閉じ、Bloch周期は不使用）、
  中央のレンズ `len(pattern) // 2` 番目（偶数枚は中央右寄り）だけを照射する。
  光源は照射レンズの真上（`coords["source_center_x_um"]`）にレンズ幅で置き、
  収束判定点も同じx位置に置く。照射レンズの画素番号は
  `coords["center_pixel_indices"]` として結果に含め、UIはこれを基準に
  「左隣／中央／右隣」を表示する。パターンは2枚以上が条件（1枚は従来の
  共有方式欄で評価）。`["single"]*3` のパターンは従来の
  「共有なし＋受光内訳」と完全一致する（回帰確認済み）

### 3.2e OCL表面の反射防止膜（F-16）

- `ocl.coat_um`（0=なし）と `materials.ocl_coat_n`（初期値1.26≈√1.58）で、
  レンズ表面を覆う薄膜を指定できる
- モデル化: レンズ面を膜厚ぶん「上へずらした」外側形状（高さ+t、
  ギャップ高さ+t）を先に膜材で描き、後から本体レンズが内側を上書きして
  殻状の薄膜が残る（縦方向オフセット近似。レンズ頂上付近では
  法線方向の膜厚≈指定値、側面が急な場所ではやや薄くなる）
- レンズ端では膜がフットプリント端まで覆い、レンズ間の谷にも膜厚ぶん残る
- 2D・3D共通の考え方で実装（build_lens_prism_2d / build_lens_slices_3d を
  高さ・ギャップ+tで呼ぶだけ）。膜厚0のときはジオメトリを追加しない
  （従来と完全一致）。両パラメータともスイープ・CSV一括計算の対象
  （2026-07-20 追加）

### 3.3 評価量の算出方法

| 評価量 | 方法 |
|---|---|
| 集光効率 | Si内フォトダイオード領域上面のフラックスモニター透過パワー ÷ 入射パワー（参照計算＝構造なし空気のみ、で規格化） |
| クロストーク・受光内訳 | 中央の共有レンズ単位のみ照射（横方向はPMLで閉じる）し、各画素PDへのフラックス ÷ 入射パワー。2Dは共有単位×3列（1画素1レンズ=3画素、2画素/4画素共有=6画素）。OCL混在パターン指定時はパターンのレンズ列を置き、中央のレンズのみ照射（3.2d参照）。3Dは3×3画素で1画素1レンズのみ対応 |
| 断面ビュー | XZ断面のDFTモニターで \|E\|² を取得 |
| 真上ビュー | 指定深さ（既定はPD面）のXY平面DFTモニターで \|E\|² を取得（3Dモードのみ） |
| CRA特性・スイープ | 入射角・波長・画素サイズ・OCL高さ・OCL曲率・CF膜厚のいずれかを外側ループでスイープし、一覧表（CSV）とグラフを出力 |
| RGB 3波長一括評価 | `rgb: true` で `rgb_wavelengths_nm`（既定 R620/G550/B450 nm）の波長スイープとして実行（result type: rgb、sweepと同形式）。表示側が色ラベルR/G/Bを付ける。スイープ・CSV一括計算とは排他（検証エラー）。ジョブ名は「RGB3波長」（2026-07-19 追加） |

### 3.4 2D／3Dモード

- 2Dモード: XZ断面の2D FDTD。数分で完了。断面の傾向把握・広いスイープ向け
- 3Dモード: フル3D。真上ビュー・共有OCL（2×2）・クロストークの定量評価向け
  - 3DのOCLは高さ方向スライス（`NUM_LENS_SLICES_3D = 20` 枚の楕円柱）で近似する
    （課題 I-009）。共有OCLは共有単位全体を覆う楕円フットプリントの1枚レンズとする
- メッシュ解像度の初期値: `RESOLUTION_PIXELS_PER_UM = 100`（=10 nm刻み）
  - Si内では波長が1/n（550 nmで約135 nm）に縮むため、空気基準のλ/20ではなく
    「Si内波長を10セル以上」で決める。フレネル検証で解像度50は誤差7.8%、
    100は2.0%、200は0.6%と収束を確認した（2026-07-12）
  - Si内波長あたり10セルを下回る条件（短波長×低解像度）では実行前に警告を出す

### 3.5 材料モデル（初期値、課題 I-003 で出典確定）

| 材料 | 屈折率の扱い |
|---|---|
| OCL樹脂 | n ≈ 1.58（分散小、定数近似から開始） |
| 平坦化膜 | n ≈ 1.50 |
| カラーフィルタ | n ≈ 1.55（吸収の扱いは課題 I-004） |
| Si | 波長分散＋吸収（k）をテーブルで保持 |
| DTI充填材 | n ≈ 1.46（SiO2想定、上書き可能） |

### 4.2b CSV一括計算（F-12）

- UIはCSVファイルをテキストとして読み（UTF-8で失敗したらShift_JISで再解釈）、
  画面の入力値JSONに `batch_csv` として添えて POST /api/jobs へ送る
- サーバーは `engine.fdtd_worker.parse_batch_csv` でケース一覧
  `{"columns": [...], "cases": [{"label", "overrides"}...]}` に変換して
  `params["batch"]` に格納し、通常のジョブとして起動する
- ワーカーはスイープと同じ仕組みで1行=1条件を順に計算する
  （画面の入力値をベースに、CSVに書かれた列だけをドット区切りパスで上書き。
  各条件は実行前に単発計算と同じ範囲チェックを通す）
- 使える列名は `BATCH_COLUMN_TYPES`（数値／1・0／選択肢文字列）と
  任意の条件名 `label`。上限は `MAX_BATCH_CASES`（100条件）
- 結果は `batch.csv`（条件名・上書き値・集光効率・画素ごとの効率、
  受光内訳ありなら中央効率とクロストーク）をExcel対応のBOM付きUTF-8で出力する
- スイープとの同時指定はエラーにする（UI側でも一括実行時は sweep を送らない）

### 4.3 ジョブ実行フロー

1. UIからパラメータJSONを受領し、入力チェック（範囲・組み合わせ）を行う
2. `jobs/<job_id>/input.json` を書き出し、Meep環境のPythonで `engine/fdtd_worker.py` を起動
3. ワーカーは進捗を `progress.json` に逐次書き出し、完了時に `result.json` / `fields.npz` を出力
4. アプリ本体が結果を読み込み、matplotlibで画像・グラフを生成して表示する

## 5. エラー処理・入力チェック方針

- パラメータ範囲外（画素サイズ 0.5〜2.5 µm 外など）は実行前にエラー表示する
- OCL高さ・曲率の組み合わせが幾何的に成立しない場合（レンズがはみ出す等）はプレビューで警告する
- ワーカー異常終了時は stderr をジョブログとして保存し、UIに「失敗」と理由を表示する
- 3Dモードで推定メモリが閾値を超える場合は実行前に警告する

## 6. 検証計画

- 平面基板（レンズなし）の透過率をフレネル係数の解析解と比較し、FDTD設定の妥当性を確認する
- 2Dと3Dで同一構造（1画素レンズ・垂直入射）の断面分布傾向が一致することを確認する
- メッシュ解像度を2倍にした際の集光効率変化が許容誤差内（目安±2%）であることを確認する

## 7. 技術選定の記録

- FDTDエンジン: **Meep**（無料・ローカル実行・実績十分）。
  Tidy3D は高速だが構造データを外部クラウドへ送信するためセキュリティルール上不採用。
  Lumerical は高額ライセンスのため不採用（導入する場合はエンジニア・購買相談）。
  MATLAB自作は開発量・検証リスクが過大なため不採用
- UI: ローカルWeb UI（FastAPI）。外部送信なし。将来のIIS社内公開（課題 I-005）にも接続しやすい
